WHAT IS MVC?

by Agata Majer

MVC stands for:


MODEL VIEW CONTROLLER

MVC is a system we use to organise our files and folders in a project.
We group our files based on their role and usage, so we'll always know where to find a specific file.
We keep the logic and the display separately.
It's basically a tree of folders that we always name similarly and MVC tells us which file goes into which folder.

First we need to understand how the internet works.

When you use the internet every time you click something or go to a new website you send a request to something called server. To simply put, servers serve content to our browsers, so they get our requests and then respond by sending specific content back to us.

In order for this process to work correctly, developers use MVC architecture to connect all the necessary files, so we don't have to worry about anything while surfing the internet.

    MVC is basically two things:
  1. LISTENING FOR REQUESTS
  2. DECIDING HOW TO RESPOND TO THE REQUESTS

MODEL

MODEL contains our app's data, which it can also change. It's the only thing that can connect to our database. It can never interact with the view.
Model is a set of objects allowing interaction with the database.
The files included in the model folder are our Schemas(templates for our content written as objects), which file names always start with a capital letter and database connection file f.ex. mongoose.

VIEW

VIEW is what the users see and interact with when they go to a specific website aka the images, menus, text, buttons etc that you can see when you open a website.
In Views folder we put our html (or any templating language file) that describes how webpages will be displayed online.
It includes code responsible for visuals (HTML/CSS/JavaScript).

CONTROLLER

CONTROLLER is the middle person in MVC. It's the thing that holds the app together and can connect to the view and the model. It's the brains and the logic of an app. It's responsible for all handling, logic etc.
It takes requests from the browser, contacts the model and takes any data that might be needed, then goes to the view to generate the website for the user to see.

MVC is like a restaurant:

In a restaurant, the Chef is the Model. It's job is to take orders (requests) from the Waiter and create some awesome food for the Waiter to deliver to the Customer.

The Waiter is the Controller. They take in orders (requests), take them back to the Chef (Model), and then deliver the proper meal to customers (Users).

The Table is the View. It just holds whatever meal the Waiter brings back from the Chef.

Why should you use MVC?

  • Makes working with others on a project together easier
  • It's easier to debug and find errors
  • When something breaks it doesn't break the entire project
  • Interchangeability
